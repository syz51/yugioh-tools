# AGENTS.md

## Project Overview

`yugioh-tools` is a single-package TanStack Start application for Yu-Gi-Oh tooling.
The current app serves a React frontend from `src/routes/index.tsx` and includes
server-side utilities for:

- parsing `.ydk` deck files
- calculating opening-hand probabilities
- caching card lookups in Postgres
- syncing the local cache from the YGOCDB upstream dataset

Core technologies in use:

- Bun for package management and script execution
- TypeScript + React 19
- TanStack Start + TanStack Router with file-based routes
- Tailwind CSS 4 for styling
- Drizzle ORM + PostgreSQL
- Vitest for tests
- Dokploy for production deployment with Bun-based Docker images stored in Tencent TCR

## Setup Commands

Install dependencies with Bun:

```bash
bun install
```

Required environment variables live in `.env` and are validated in `src/env.ts`:

- `DATABASE_URL` is required for any server/database work
- `SERVER_URL` is optional
- `VITE_APP_TITLE` is an optional client variable

Start the development server:

```bash
bun --bun run dev
```

Useful project commands:

```bash
bun --bun run build
bun --bun run preview
bun --bun run start
bun --bun run test
bun --bun run lint
bunx tsc --noEmit
bun --bun run format
bun --bun run check
bun --bun run db:generate
bun --bun run db:migrate
bun --bun run db:studio
bun --bun run db:push
```

## Development Workflow

### Routing and app structure

- File-based routes live in `src/routes`.
- `src/routeTree.gen.ts` is generated by TanStack Router. Do not edit it by hand.
- Shared UI lives under `src/components`.
- Domain logic and server helpers live under `src/lib`.
- Database access lives under `src/db`.

### Server/client boundaries

- Keep database access and upstream network fetches in server-only modules.
- Existing server-only files import `@tanstack/react-start/server-only`; follow that pattern for new server modules.
- API handlers currently live as TanStack file routes, for example `src/routes/api/ygocdb/sync.ts`.

### Database and migrations

- Drizzle schema source of truth is `src/db/schema.ts`.
- Generated SQL migrations live in `drizzle/`.
- If you change the schema, update the TypeScript schema and generate a new Drizzle migration rather than editing old migrations in place.
- Commit the generated SQL and Drizzle metadata alongside schema changes.

### UI changes

- Styling is Tailwind-first via `src/styles.css`.
- The repo includes `components.json`; if you add shadcn components, use the latest CLI:

```bash
pnpm dlx shadcn@latest add <component>
```

- When installing shadcn components, prefer the latest CLI invocation above instead of older templates or copied component code.

## Testing Instructions

- Test files are colocated with source and use `*.test.ts`.
- Run the full suite before finishing substantial code changes:

```bash
bun --bun run test
bunx tsc --noEmit
```

- You can run a focused test file with Vitest directly:

```bash
bunx vitest run src/lib/ydk.test.ts
```

- Prefer adding or updating tests when touching parsing logic, probability logic, sync behavior, or route handlers.

## Code Style

- TypeScript uses ESM and the `#/*` import alias for `src/*`.
- Formatting is Prettier with:
  - no semicolons
  - single quotes
  - trailing commas
- ESLint is based on `@tanstack/eslint-config` with a few disabled rules in `eslint.config.js`.
- Follow the existing file-local style: concise functions, explicit types around shared boundaries, and colocated tests.
- Avoid editing generated output such as `src/routeTree.gen.ts`, `dist/`, or `.output/` artifacts unless the task is specifically about generated output.

## Build and Deployment

- Local production build:

```bash
bun --bun run build
```

- TanStack Start builds the production artifact into `.output/`.
- The production container is built from `Dockerfile` and runs `bun .output/server/index.mjs`.
- Dokploy should deploy the image pushed by `.github/workflows/deploy.yml`.
- Production database migrations are triggered through Dokploy webhooks from `.github/workflows/deploy.yml`.
- `.github/workflows/migrate-production.yml` is a manual fallback that calls the Dokploy migration webhook directly.

## PR and Change Guidance

- Run `bun --bun run test`, `bun --bun run lint`, and `bunx tsc --noEmit` before finalizing non-trivial changes.
- If you change formatting broadly, be explicit about it because `bun --bun run check` rewrites files with Prettier and ESLint autofixes.
- Keep changes scoped. The repo is small and tests are fast, so prefer updating the closest test file when behavior changes.
- Preserve generated-vs-source boundaries: edit the source route/schema/component files, then regenerate derived artifacts if needed.

## Project-Specific Notes

- `src/lib/ygocdb-sync.server.ts` downloads `cards.zip` from YGOCDB, validates the MD5, and replaces the cached card table transactionally.
- `src/lib/ygocdb-cache.ts` reads from Postgres first and falls back to upstream fetches for missing cards.
- `src/lib/ydk.ts` contains deck parsing and deck-construction validation helpers.
- `src/lib/opening-hand-calculator.ts` contains the probability logic; keep changes there covered by tests because correctness matters more than micro-optimizations.
